# Docusaurus Documentation + npm/PyPI Packaging

## Summary

Replace mdBook with Docusaurus for the documentation site, ship npm packages via auto-download binary wrapper, and ship a PyPI wheel via maturin — all in one branch. Schemalint is a JS/TS + Python tool whose engine happens to be written in Rust.

---

## Problem Frame

Schemalint's primary audience is JavaScript/TypeScript and Python developers. Today those developers cannot install it without a Rust toolchain (`cargo install`). The npm packages don't exist, PyPI has no wheel, and the documentation site is built with mdBook — a Rust ecosystem tool foreign to the target audience. The project reads as "Rust tool with helpers" when it should read as "cross-platform linter that happens to use Rust for performance."

The existing `docs/book/` mdBook site (Phase 5 U4) works but burns engineering effort on a docs framework your users don't recognize. Docusaurus is the standard for JS/TS-facing projects. The existing `typescript/schemalint-zod/` package proves the project already has a TypeScript footprint — it needs consistent distribution infrastructure alongside it.

---

## Actors

- A1. **TypeScript/JavaScript developer**: Installs via npm, reads Docusaurus docs, integrates schemalint into Node.js project toolchains.
- A2. **Python developer**: Installs via pip, reads Docusaurus docs, integrates schemalint into Pydantic project toolchains.
- A3. **CI system**: Downloads binaries from GitHub Releases, installs via npm/pip in CI pipelines, consumes docs in CI-friendly format.
- A4. **Documentation reader**: Browses the Docusaurus site for installation guides, rule reference, and configuration docs.

---

## Key Flows

- F1. **Install and run via npm**
  - **Trigger:** `npm install -g @schemalint/cli`
  - **Actors:** A1
  - **Steps:** npm installs the CLI package → package runs JS entry point → JS detects platform, downloads the matching GitHub Release binary, caches it → binary runs with forwarded args
  - **Outcome:** `schemalint --version` prints version, `schemalint check` works without Rust toolchain
  - **Covered by:** R5, R7, R10

- F2. **Install and run via pip**
  - **Trigger:** `pip install schemalint`
  - **Actors:** A2
  - **Steps:** pip downloads wheel → wheel contains bundled schemalint binary → binary placed on PATH → user runs `schemalint check` 
  - **Outcome:** `schemalint --version` prints version, `schemalint check` works without Rust toolchain
  - **Covered by:** R4, R6, R10

- F3. **Browse documentation**
  - **Trigger:** User navigates to docs URL or runs `npm start` in docs directory
  - **Actors:** A1, A2, A4
  - **Steps:** Docusaurus site loads → navigation shows installation, quick start, configuration, profiles, and rule reference → rule reference pages are auto-generated from compile-time rule metadata → user searches for specific rules
  - **Outcome:** Developer finds install instructions, rule reference, and configuration docs in an expected JS/TS docs framework
  - **Covered by:** R1, R2, R3

## Requirements

**Docusaurus documentation site**
- R1. Docusaurus site replaces mdBook entirely — old `docs/book/` directory removed, new `docs/` directory is the Docusaurus project root
- R2. All hand-written documentation content (installation, quick-start, configuration, CI integration, profiles) migrated from mdBook markdown to Docusaurus MDX/docs format
- R3. Rule reference pages auto-generated by `schemalint-docgen` into Docusaurus-compatible markdown files, committed to the repo for local `npm start` without running docgen

**PyPI packaging**
- R4. `crates/schemalint-python/` crate with maturin `bindings = "bin"` — thin wrapper re-exporting `schemalint::cli::run()`
- R5. `maturin build --release` produces a `.whl` file; `pip install` places `schemalint` on PATH
- R6. Wheel targets: macOS x86_64, macOS aarch64, Linux x86_64 (manylinux), Linux aarch64 (manylinux), Windows x86_64

**npm packaging**
- R7. `@schemalint/cli` npm package at `npm/cli/` — JS entry point downloads platform-appropriate binary from GitHub Releases on first run, caches locally, verifies SHA256, execs binary
- R8. Existing `@schemalint/zod` package at `typescript/schemalint-zod/` published to npm alongside `@schemalint/cli`
- R9. `@schemalint/core` placeholder package at `npm/core/` — requires `schemalint` CLI on PATH, defers napi-rs native bindings to follow-up

**Quality and verification**
- R10. Smoke tests on every platform: install from each channel → `schemalint --version` → `schemalint check` on a known-good schema produces expected output
- R11. `cargo test --workspace` passes, `cargo clippy --workspace -- -D warnings` clean, `cargo fmt --all -- --check` clean

**CI and deployment**
- R12. GitHub Actions workflow deploys Docusaurus site to GitHub Pages on push to main
- R13. Docusaurus staleness check in CI: after docgen runs, `git diff --exit-code` on docs directory fails if auto-generated pages are stale

## Acceptance Examples

- AE1. **Covers R7, R10.** Given a macOS x86_64 machine with Node.js, `npm install -g @schemalint/cli` downloads cached binary; `schemalint check --profile openai.so.2026-04-30 schema.json` exits 0 for a clean schema.
- AE2. **Covers R4, R5, R10.** Given a Linux x86_64 machine with Python 3.8+, `pip install schemalint` installs the CLI; `schemalint --version` reports the installed version; `schemalint check` on a forbidden-keyword schema exits 1.
- AE3. **Covers R1, R2, R3.** Given a clean checkout, `npm install && npm start` in the docs directory serves the Docusaurus site locally with working navigation to installation, quick start, configuration, profiles, and auto-generated rule reference pages.
- AE4. **Covers R12, R13.** Given a push to main with rule changes, CI runs docgen, detects stale auto-generated rule pages, fails the check with clear instruction to run docgen locally.

## Success Criteria

- A JS/TS developer can `npm install -g @schemalint/cli` and run `schemalint check` without ever installing Rust
- A Python developer can `pip install schemalint` and run `schemalint check` without ever installing Rust
- The documentation site looks native to the target audience — Docusaurus, not mdBook
- Every piece of content from the Phase 5 mdBook site is preserved and accessible in the Docusaurus site
- `cargo test --workspace` passes with zero regressions

---

## Scope Boundaries

- Full marketing landing page — docs site only for this phase
- napi-rs native Node.js module — binary download wrapper only
- `@schemalint/core` as a real programmatic API — placeholder requiring CLI on PATH
- Homebrew formula, Docker distribution, or other channels
- Changing the Rust core engine, rule engine, or CLI behavior
- Provider-side conformance, auto-fix, or LSP integration

---

## Key Decisions

| Decision | Rationale |
|---|---|
| Docusaurus over mdBook | Target audience is JS/TS + Python developers; Docusaurus is the standard docs framework they expect |
| Auto-download binary wrapper over napi-rs | Ships fastest. napi-rs native module adds complexity surface (build matrix, platform packages, N-API versioning) without proportional user value for a CLI tool |
| maturin `bindings = "bin"` over pyo3 library bindings | CLI-only distribution — no Python API surface needed. `bindings = "bin"` is simpler and sufficient |
| Docs Docusaurus at repo root over subdirectory | Standard Docusaurus convention; clearer than nesting under `docs/book/` |
| Everything in one branch | The three pieces (Docusaurus, npm, PyPI) form a coherent launch. Separate branches create sequencing dependency risk with no benefit |

## Dependencies / Assumptions

- GitHub Releases exist for the current version before npm binary download works — release workflow (Phase 5 U14, deferred) must publish first
- `@schemalint` npm scope is available and registered to the 1nder-labs org
- `schemalint` name is available on PyPI
- Environment secrets (`NPM_TOKEN`, `MATURIN_PYPI_TOKEN`) configured in GitHub Actions
- Docusaurus deployed to GitHub Pages; repo Settings → Pages set to "GitHub Actions"
- Existing `docs/book/src/` markdown content is directly portable to Docusaurus — Docusaurus renders standard markdown natively
