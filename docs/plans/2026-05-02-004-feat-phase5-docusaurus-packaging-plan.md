# feat: Phase 5 — Docusaurus Docs + npm/PyPI Packaging

## Summary

Replace the mdBook documentation site with Docusaurus, strip all mdBook artifacts, ship `@schemalint/cli` to npm via auto-download binary wrapper, publish the existing `@schemalint/zod`, create a `@schemalint/core` placeholder, and package the CLI binary as a PyPI wheel via maturin — all in one branch from `feat/phase5-distribution-conformance`. The Rust core is the build layer; the product ships to JS/TS and Python developers.

---

## Problem Frame

The target audience — JS/TS and Python developers — cannot install schemalint without a Rust toolchain. The docs are built with mdBook, a Rust ecosystem tool foreign to that audience. This branch strips the Rust-only identity and repositions schemalint as a cross-platform tool distributed via the channels its users actually use: npm and pip.

---

## Requirements

**Docusaurus documentation site**
- R1. Docusaurus site replaces mdBook entirely — `docs/book/` and its artifacts removed, new Docusaurus site lives under `docs/`
- R2. All hand-written content (installation, quick-start, configuration, CI integration, profiles) migrated from mdBook markdown to Docusaurus docs
- R3. Rule reference pages auto-generated by `schemalint-docgen` into a Docusaurus docs directory, committed to repo

**PyPI packaging**
- R4. `crates/schemalint-python/` crate with maturin `bindings = "bin"` — thin wrapper re-exporting `schemalint::cli::run()`
- R5. `maturin build --release` produces a `.whl`; `pip install schemalint` places `schemalint` on PATH
- R6. Wheel targets: macOS x86_64/aarch64, Linux x86_64/aarch64 (manylinux), Windows x86_64

**npm packaging**
- R7. `@schemalint/cli` at `npm/cli/` — JS entry point auto-downloads platform binary from GitHub Releases, caches locally, verifies SHA256, forwards args
- R8. Existing `@schemalint/zod` at `typescript/schemalint-zod/` published to npm alongside `@schemalint/cli`
- R9. `@schemalint/core` placeholder at `npm/core/` — requires `schemalint` CLI on PATH, defers napi-rs to follow-up

**CI and quality**
- R10. Smoke tests per platform (install → version check → known-good schema check)
- R11. `cargo test --workspace` passes, clippy clean, fmt clean
- R12. GitHub Actions deploys Docusaurus to GitHub Pages on push to main
- R13. Docusaurus staleness check in CI: docgen run → `git diff --exit-code` on docs dir

**Origin actors:** A1 (TypeScript/JavaScript developer), A2 (Python developer), A3 (CI system), A4 (Documentation reader)

## Context & Research

### Relevant Code and Patterns

- **Workspace structure** (4 crates): `schemalint` (core + CLI), `schemalint-profiles` (TOML data), `schemalint-docgen` (rule doc generator, `publish = false`), `schemalint-conformance` (truth engine + mock server, `publish = false`) — `Cargo.toml`
- **CLI entry point**: `crates/schemalint/src/main.rs` → `schemalint::cli::run()` — 3-line entry
- **Docgen binary**: `crates/schemalint-docgen/src/main.rs` — CLI args `--output-dir` (default: `docs/book/src/rules`), `--summary-path` (default: `docs/book/src/SUMMARY.md`). Generates per-rule `.md` pages into category dirs, injects links into `SUMMARY.md` between `<!-- AUTO-GENERATED RULES -->` markers
- **Existing npm package**: `typescript/schemalint-zod/package.json` — `schemalint-zod` v0.1.0, `bin` entry `schemalint-zod`, ESM exports, `tsc` build
- **Existing Python package**: `python/schemalint-pydantic/pyproject.toml` — `schemalint-pydantic` v0.1.0, setuptools build, JSON-RPC server entrypoint
- **CI patterns**: `ci.yml` — `dtolnay/rust-toolchain@stable`, `Swatinem/rust-cache@v2`, `actions/setup-node@v4` with Node 22. `docs.yml` — `peaceiris/actions-mdbook@v2` + `upload-pages-artifact@v4` + `deploy-pages@v5`
- **Rule metadata**: `RuleMetadata` struct with `name`, `description`, `rationale`, `severity`, `category` (`Keyword`/`Restriction`/`Structural`/`Semantic`), `bad_example`, `good_example`, `see_also`, `profile` — `crates/schemalint/src/rules/metadata.rs`
- **Error codes**: `{prefix}` placeholder pattern (expanded to `OAI`/`ANT` per profile) — used by docgen
- **`publish = false` convention**: internal-only crates (docgen, conformance) use `publish = false`
- **Output emitter pattern**: Each format in a separate `cli/emit_*.rs` file, shared `emit_X_to_string(diagnostics, ...) -> String` signature
- **Subprocess helper pattern**: JSON-RPC 2.0 over stdin/stdout, configurable timeout — `node/mod.rs`, `python/mod.rs`
- **esbuild auto-download pattern**: industry standard for CLI tools distributed via npm — download platform binary on first run, cache locally, verify checksum

### Institutional Learnings

- **GHA workflow command escaping** (`docs/solutions/best-practices/schemalint-phase2-learnings.md`): Any new scheduled workflow output must use `percent_encoding` for `::error`/`::warning` commands. Reference implementation: `cli/emit_gha.rs` `encode_gha_value()`.
- **Test field name matching**: Tests asserting structured output must use emitter JSON field names (e.g., `pointer`), not internal Rust struct field names (e.g., `schema_path`).
- **I/O error logging**: Use `if let Err(e) = ... { eprintln!("warning: {e}"); }` instead of `let _ =` for non-critical paths.
- **Pipeline dedup**: `run_check` and `handle_check` in `cli/mod.rs` duplicate ~30 lines of pipeline orchestration. Do not add a third copy.

### External References

- **Docusaurus v3**: `npx create-docusaurus@latest` initializes project; `docusaurus.config.js` for config; `sidebars.js` or autogenerated sidebars; standard markdown in `docs/` directory; `@docusaurus/preset-classic` for docs-only site
- **maturin** (v1+): PEP 621 `pyproject.toml`; `[tool.maturin] bindings = "bin"` for CLI-only crates; `PyO3/maturin-action@v1` for CI; `--compatibility pypi` for pre-upload validation
- **npm binary download pattern**: `package.json` with `bin` field; JS entry point detects platform via `os`/`arch`, downloads binary from GitHub Releases URL, extracts to cache dir, exec's with forwarded args; SHA256 verification
- **GitHub Pages deployment**: `actions/upload-pages-artifact@v4` + `actions/deploy-pages@v5`

## Key Technical Decisions

| Decision | Rationale |
|---|---|
| Docusaurus v3 with standard `.md` files, not MDX | Existing mdBook content is plain markdown — `.md` avoids format conversion. MDX adds complexity (React components in docs) with no proportional value for a rule reference site |
| Docgen writes to `docs/rules/` at Docusaurus root, not `docs/` subdirectory | Keeps auto-generated content in a clean subdirectory separate from hand-written docs. Docusaurus `sidebars.js` references it |
| Docgen drops SUMMARY.md injection, generates Docusaurus-compatible sidebar instead | mdBook's `SUMMARY.md` format is not compatible with Docusaurus. Generating `sidebars.js` rules section or `_category_.json` files is the correct Docusaurus pattern |
| maturin `bindings = "bin"` in a new `schemalint-python` crate | Maturin "bin" mode bundles a Rust binary as a Python wheel entry point. New crate avoids bloating the main `schemalint` crate with Python-specific config |
| npm auto-download binary from GitHub Releases | Simplest to ship. Binary is platform-specific; JS wrapper handles detection and download. Consistent with esbuild, swc, Prettier patterns |
| Docusaurus site lives at `docs/` (repo root level) | Standard Docusaurus convention. Old `docs/book/` removed entirely. `docs/solutions/` and `docs/plans/` preserved alongside new Docusaurus structure |
| `@schemalint/core` as bare placeholder | R7 says "programmatic API" but napi-rs is deferred. Placeholder ships the intent, real API follows with napi-rs |
| `@schemalint/zod` version syncs from git tag, not manual version | Keeps all npm packages on the same version. Enforced in CI/release workflow |
| No GitBook-style `docs/` subdirectory — Docusaurus project IS the docs directory | Avoids `docs/website/` nesting. `docusaurus.config.js`, `sidebars.js`, `src/`, and `docs/` (Docusaurus content) all live under `docs/` |

- U3. **Initialize Docusaurus site and migrate content**

**Goal:** Set up a new Docusaurus v3 project at `docs/`, configure it as a docs-only site, migrate all hand-written content, and integrate auto-generated rule pages into the sidebar.

**Requirements:** R1, R2, R3

**Dependencies:** U1, U2

**Files:**
- Create: `docs/docusaurus.config.js`
- Create: `docs/sidebars.js`
- Create: `docs/package.json`
- Create: `docs/babel.config.js` (if needed by Docusaurus v3)

- Modify: `docs/docs/profiles/*.md` (update cross-references)
- Create: `docs/src/pages/index.js` (homepage redirect or custom landing)
- Create: `docs/static/.gitkeep`

**Approach:**
- Initialize Docusaurus v3 manually (not via `create-docusaurus` scaffolding — too many unused files). Create minimal config:
  ```js
  // docusaurus.config.js
  module.exports = {
    title: 'Schemalint',
    tagline: 'Static analysis for JSON Schema compatibility with LLM structured-output providers',
    url: 'https://1nder-labs.github.io',
    baseUrl: '/schemalint/',
    organizationName: '1nder-labs',
    projectName: 'schemalint',
    presets: [['@docusaurus/preset-classic', {
      docs: { path: 'docs', routeBasePath: '/', sidebarPath: './sidebars.js' },
      blog: false,       // docs-only, no blog
      theme: { customCss: './src/css/custom.css' }
    }]],
  };
  ```
- Set `routeBasePath: '/'` so docs are served at root — the entire site IS the documentation
- `sidebars.js` includes both hand-written sidebar entries AND auto-generated `rules/` entries:
  ```js
  module.exports = {
    docsSidebar: [
      'index',
      { type: 'category', label: 'Guide', items: ['guide/installation', 'guide/quick-start'] },
      { type: 'category', label: 'Configuration', items: ['guide/configuration', 'guide/ci-integration'] },
      { type: 'category', label: 'Profiles', items: ['profiles/index', 'profiles/openai', 'profiles/anthropic'] },
      // Rules section is generated by docgen — sidebar references rules/index + auto-generated pages
      { type: 'category', label: 'Rule Reference', items: [{ type: 'autogenerated', dirName: 'rules' }] },
    ],
  };
  ```
  Using `{ type: 'autogenerated', dirName: 'rules' }` means Docusaurus automatically picks up all `.md` files under `docs/docs/rules/` and structures them by directory — the `_category_.json` files generated by docgen control labels and ordering.
- `package.json` with Docusaurus v3 deps: `@docusaurus/core`, `@docusaurus/preset-classic`, `react`, `react-dom`
- Homepage at `src/pages/index.js` can be a simple redirect to `/` or a custom React component
- All cross-reference links updated from mdBook style (`[text](./page.md)`) to Docusaurus style (same format, Docusaurus handles `.md` → route resolution)
- Do NOT use MDX — standard `.md` only for simplicity and direct migration

**Patterns to follow:** Docusaurus v3 docs-only config pattern. Standard Docusaurus `package.json` structure with `scripts: { start, build, serve }`.

**Test scenarios:**
- Happy path: `cd docs && npm install && npm start` serves the site locally
- Happy path: Navigation sidebar shows Guide, Configuration, Profiles, and Rule Reference sections
- Happy path: All cross-reference links between pages resolve correctly
- Happy path: Rule reference pages are reachable via sidebar auto-generation

**Goal:** Create a new workspace crate `schemalint-python` that uses maturin to bundle the schemalint CLI binary as a Python wheel.

**Requirements:** R4, R5

**Dependencies:** None (new crate, depends on `schemalint` as path dependency)

**Files:**

- Create: `crates/schemalint-python/pyproject.toml`
- Create: `crates/schemalint-python/src/main.rs`
- Create: `crates/schemalint-python/python/schemalint/__init__.py`
- Create: `crates/schemalint-python/python/schemalint/cli.py`

**Approach:**
- `Cargo.toml`: workspace member, `publish = false` (published via maturin, not cargo), depends on `schemalint` (path dep):
  ```toml
  [package]
  name = "schemalint-python"
  version.workspace = true
  edition.workspace = true
  license.workspace = true
  publish = false

  [[bin]]
  name = "schemalint"
  path = "src/main.rs"

  [dependencies]
  schemalint = { path = "../schemalint" }
  ```
- `pyproject.toml`: PEP 621 metadata with maturin config:
  ```toml
  [build-system]
  requires = ["maturin>=1.8,<2.0"]
  build-backend = "maturin"

  [project]
  name = "schemalint"
  version = "0.1.0"
  description = "Static analysis for JSON Schema compatibility with LLM structured-output providers"
  readme = "../../README.md"
  license = { text = "MIT OR Apache-2.0" }

  [tool.maturin]
  bindings = "bin"
  python-source = "python"
  manifest-path = "Cargo.toml"
  ```
- `src/main.rs`: Thin wrapper that calls the existing CLI entry point:
  ```rust
  fn main() {
      schemalint::cli::run();
  }
  ```
- `python/schemalint/__init__.py`: Empty or version-only
- `python/schemalint/cli.py`: Minimal shim that the wheel entry point can reference:
  ```python
  import subprocess, sys
  def main():
      # The binary is installed as a console_scripts entry point by maturin
      pass
  ```
  Actually — maturin's `bindings = "bin"` creates the entry point from the `[[bin]]` target name automatically. The Python files are needed only if we want `python-source` to expose additional Python code. For a pure binary distribution, `__init__.py` and `cli.py` are minimal — maturin handles the `console_scripts` entry point.

**Patterns to follow:** Official maturin `bindings = "bin"` documentation. Existing workspace member conventions (`version.workspace = true`, `edition.workspace = true`). Existing `publish = false` convention for internal-only crates.

**Test scenarios:**
- Happy path: `maturin build --release` in `crates/schemalint-python/` produces a `.whl` file
- Happy path: `pip install target/wheels/schemalint-*.whl` installs the CLI; `schemalint --version` works
- Happy path: Installed binary runs `schemalint check --profile openai.so.2026-04-30 schema.json` on known-good schema → exits 0
- Edge case: Wheel installed on Python 3.8 (oldest supported) — CLI binary is platform-native, not Python-version-dependent
- Error path: Attempting `maturin build` without schemalint compiled → clear build error

**Verification:**
- `cargo build --workspace` compiles `schemalint-python`

- U7. **Create npm CLI auto-download wrapper**

**Goal:** Create `@schemalint/cli` npm package that downloads the platform-appropriate schemalint binary from GitHub Releases and exec's it with forwarded args.

**Requirements:** R7

**Dependencies:** None (GH Releases URL is hardcoded to a known version)

**Files:**
- Create: `npm/cli/package.json`
- Create: `npm/cli/index.js`
- Create: `npm/cli/install.js` (postinstall script — optional, might download on install)

**Approach:**
- `package.json`:
  ```json
  {
    "name": "@schemalint/cli",
    "version": "0.1.0",
    "description": "schemalint CLI — static analysis for JSON Schema LLM compatibility",
    "bin": { "schemalint": "./index.js" },
    "files": ["index.js", "install.js"],
    "scripts": { "postinstall": "node install.js" }
  }
  ```
- `index.js` — main entry point:
  1. Detect platform + arch via `process.platform` / `process.arch`
  2. Map to cargo-dist target triple (e.g., `darwin-x64` → `x86_64-apple-darwin`)
  3. Get version from `package.json` or a fixed version
  4. Construct download URL: `https://github.com/1nder-labs/schemalint/releases/download/v{version}/schemalint-{target}.tar.gz`
  5. Download to `~/.cache/schemalint-npm/{version}/{target}/`, extract, cache path
  6. On subsequent runs, use cached binary if checksum matches
  7. `child_process.spawnSync(binaryPath, process.argv.slice(2), { stdio: 'inherit' })` — forward exit code
  8. If binary not found on any platform: exit with clear error listing supported platforms
- `install.js` — optional postinstall: pre-download the binary for the current platform
- SHA256 verification: compare downloaded binary hash against a hardcoded checksum or a checksum file fetched alongside the binary
- Cache invalidation: version + platform + arch as cache key

**Patterns to follow:** esbuild npm distribution pattern. `child_process.spawnSync` with `stdio: 'inherit'` for clean exit code forwarding. `os.arch()` mapping table. `fs.mkdirSync({ recursive: true })` for cache dir creation.

**Test scenarios:**
- Happy path: `node index.js --version` downloads binary (if not cached), prints version
- Happy path: Second invocation uses cached binary — no network request
- Edge case: Unknown platform (e.g., FreeBSD) → clear error with supported platform list
- Edge case: Binary download fails (network error) → retry with backoff, clear error on exhaustion
- Error path: GitHub Releases binary not published for current version → clear error message
- Error path: SHA256 checksum mismatch → blocks execution, clear error

**Verification:**
- `node npm/cli/index.js --version` downloads and runs the binary (when GH Release exists)
- `node npm/cli/index.js check --profile openai.so.2026-04-30 schema.json` forwards args and exit code
- Cached binary path: `~/.cache/schemalint-npm/0.1.0/{platform}/schemalint`

  ```json

    "version": "0.1.0",
    "description": "schemalint programmatic API — requires schemalint CLI on PATH",
    "main": "index.js",
    "files": ["index.js"]
  }
  ```
- `index.js`: Exports a `lint()` function that spawns `schemalint check --format json` and parses stdout. If `schemalint` is not on PATH, throws a descriptive error.
  ```js
  const { spawnSync } = require('child_process');

  function lint(schemaPath, options = {}) {
    const args = ['check', '--format', 'json'];
    if (options.profile) args.push('--profile', options.profile);
    args.push(schemaPath);

    const result = spawnSync('schemalint', args, { encoding: 'utf-8' });
    if (result.error && result.error.code === 'ENOENT') {
      throw new Error('schemalint CLI not found on PATH. Install via: npm install -g @schemalint/cli');
    }
    try {
      return JSON.parse(result.stdout);
    } catch {
      return { issues: [], error: result.stderr || 'unknown error' };
    }
  }

  module.exports = { lint };
  ```

**Patterns to follow:** Existing Phase 4 `lint()` function pattern (subprocess-based). Error message style from existing subprocess helpers (`node/mod.rs`, `python/mod.rs`).

**Test scenarios:**
- Happy path: `lint('./schema.json', { profile: 'openai.so.2026-04-30' })` returns parsed JSON diagnostics when `schemalint` is on PATH
- Edge case: `schemalint` not on PATH → throws descriptive error with install instructions
- Error path: `schemalint` process crashes → returns error object from stderr

**Verification:**
- `node -e "require('./index.js').lint('test-schema.json')"` runs when CLI is on PATH

## System-Wide Impact

- **Interaction graph:** The new `schemalint-python` crate depends on `schemalint` (path dep) but has no runtime interaction with the CLI binary — it IS the CLI binary, just bundled differently. The npm packages are pure JavaScript, no Rust dependencies. Docusaurus is a static site, no Rust interaction.
- **Error propagation:** npm binary download failures surface as Node.js errors with clear messages. PyPI installation failures surface at the pip level. Docgen failures surface at CI (`docgen-check` job) and build time.
- **State lifecycle risks:** npm binary cache at `~/.cache/schemalint-npm/` — stale cache risk if version pinning is incorrect. Mitigated by SHA256 verification and version-keyed cache paths.
- **API surface parity:** No Rust API changes. The `schemalint` library's public API is unchanged. Docgen output directory changes are backwards compatible via CLI args.
- **Integration coverage:** Docusaurus docs serve hand-written content + auto-generated rules. npm auto-download wrapper exec's the same binary tested in CI. PyPI wheel contains the same binary.
- **Unchanged invariants:** Existing CLI behavior, rule checking, output formats, and corpus tests are not modified. All 80+ corpus schemas continue to pass. The `schemalint-profiles` crate remains zero-dependencies.

## Risks & Dependencies

| Risk | Mitigation |
|------|------------|
| `@schemalint` npm scope not available | Fall back to unscoped `schemalint-cli` or `@1nder-labs/schemalint-cli` |
| `schemalint` name taken on PyPI | Use `schemalint-cli` or `schemalint-tool` |
| Docusaurus build output conflicts with existing `docs/` content | Use `.gitignore` to exclude `docs/build/`, `docs/.docusaurus/`, `docs/node_modules/` |
| npm auto-download depends on GitHub Releases that don't exist yet | Hardcode a known version; first release creates the binaries. Downgrade npm wrapper to print "no binary for this platform yet" gracefully |
| maturin cross-compilation for aarch64/manylinux fails in CI | Smoke-test only x86_64 Linux in CI. Full target matrix deferred to release workflow |
| Docgen SUMMARY.md removal breaks the CI staleness check | U2 replaces SUMMARY injection with `_category_.json`. U4 updates docs.yml to check Docusaurus paths |
| Docusaurus site URL (`/schemalint/`) conflicts with repo name | Configurable via `baseUrl` in `docusaurus.config.js` — matches existing mdBook site-url |
